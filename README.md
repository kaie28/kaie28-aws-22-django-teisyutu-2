### 1.【公開】

・AWSとDjangoのソースコードは、このGitHubに公開しています。

・webアプリ環境は、現在閉鎖中です。

### 2【概要】

・シンプルなTODOアプリを作成し、TerraformやDockerを用いた「AWSの自動化」と「セキュリティ対策」を中心に学習しました。

・本番環境でのデプロイに関する補足として、安全に実現するためにSecurity Group、IAM、およびAWS Systems Manager（SSM）を連携させた、セキュアなSSH（22番）接続モデルを設計・構築しました。


### 3【開発背景】

・趣味の旅行で愛用するInstagramの裏の仕組みに惹かれ、構築を学びました。手動運用の多さを知る中で、前職の看護師時代に痛感した人手不足と共通する「効率化への課題」を実感しました。今後はAWSの自動化を推進して運用負荷を減らし、新たな開発に挑戦するのが目標です。

・同アプリで実際に使われている技術や仕組みを参考に、プログラミング未経験から5ヶ月での完成を目指し、学習の順序や開発範囲を入念に計画しました。構築の際は、特に「セキュリティ強化」「データ復元の簡易化」「コスト面」に工夫を凝らし、設計からインフラ構築にいたる全ての工程を独学で実装しました。



### 4【工夫した点】

・本番環境においてはDockerを用いたコンテナ運用を行い、同時に、本番稼働中であってもリアルタイムでログからエラーを追跡・検知できる監視設計を組み込みました。

・シークレットキー等の接続情報などの機密情報を.env および .gitignore を用いて環境変数として外部管理することで、GitHub経由の認証情報漏洩リスク対策をしました。

・当初の手動構築におけるデータ消失を機に、環境をすぐ再現できる仕組みの確保が不可欠だと痛感しました。そこで、Dockerでのコンテナ化とTerraformによる自動化を採用し、万が一の際にも即座に復元できる「障害耐性の高い環境」へと強化しました。

【DBの注意】開発/本番環境は共に「db.sqlite3」を使用していますが、本番環境ではDockerのボリューム機能(コンテナ外の安全金庫)である「db_volume」と紐づけています。なので、コンテナの再起動や予期せぬ停止が発生しても「文字・CSS・画像」は消えず、安全に保持されます。(※計画時の目標に、「WebサーバーとDjango連携を最速で学習する」を優先したため、この形式にしました)。
